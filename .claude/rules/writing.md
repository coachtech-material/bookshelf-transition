# 執筆ルール

## 1. 文体

### トーン

- **本文**: 落ち着いた敬体で丁寧に解説する。過度に個人的な語りかけ（「率直に言うと」等）は避ける
- **🧠 先輩エンジニアの思考プロセス**: 実体験ベースの語りをここに集中させる。本文より口語的な表現を許容するが、「率直に言うと」「正直に言うと」等の前置きや読者を試す表現は避け、具体的な失敗談・設計判断の裏話・実務のあるあるで語る

### 用語・表記

- 敬体（です・ます調）で統一。読者を指す語は「あなた」「学習者」を用い、特定の制度名・組織名（受講区分・所属組織名など）は本文で使わない
- 「旧教材／新教材」「差分」「Bookshelf」は本文で使わない。**採点に関する表現（合格・成績・配点・点数・採点ライン等）も本文に一切出さない**。旧側は「2026年2月以前の教材」、新側は「2026年3月以降の教材」、テストは「新しい模擬案件」と表現する。運営側の背景・事情は本文に書かない（詳細は CLAUDE.md「表現の方針」）
- ダッシュ記号（`——`、`—`、`–`）は使わない
- Laravel の用語は Laravel 公式ドキュメント（10.x）の表記に従う。範囲外の内容は WebSearch / WebFetch で裏付ける
- 時間経過で変わる情報は最新情報を取得し「○○時点」と日付を明記
- 教材が前提とするバージョンと、読者が現場で遭遇しうるバージョンに差がある場合は、本文は前提バージョンで書き、差分は 💡 / 📝 の短いコラムで補足する（本文を散らかさない）
- 外部ソースを参照する場合は出典を明記する
- `**太字**` は語句・キーフレーズに限定する。文全体を太字にしない。💡 / 🔑 / 📝 プレフィックス付きの行でも同様（❌ `💡 **○○は△△です。**` → ✅ `💡 ○○は **△△** です。`）
- Markdown の `**text**` は閉じ `**` 直後にスペースなしで日本語が続くと適用されない。`**○○** の場合` のようにスペースを入れる。全角括弧 `（` も同様: `**○○**（補足）` → `**○○** （補足）`

| 表記 | 備考 |
|---|---|
| セクション | 教材本文ではカタカナ表記。内部ドキュメントでは英語 |
| Eloquent / Blade / Fortify / Sail / Policy / Gate / FormRequest / Resource / Tinker | クラス名・固有名詞は英語のまま（カタカナ化しない） |
| マイグレーション / シーダー / ファクトリ / バリデーション / ルーティング / コントローラ / ミドルウェア / リレーション / ピボットテーブル | カタカナ表記 |
| Eager Loading / N+1 問題 | 英語表記（N+1 はそのまま） |
| 認証 / 認可 | 漢字で統一（authentication / authorization） |

### 絵文字

| 絵文字 | 用途 |
|---|---|
| 🎯 | 学習目標（Section 冒頭） |
| ✨ | まとめ（Section 末尾） |
| 💡 | TIP・補足 |
| ⚠️ | 注意事項・よくあるエラー |
| 📝 | ノート・定義・補足説明 |
| 🔑 | 重要概念・キーポイント |
| 🏃 | ハンズオン手順・実践 |
| 🧠 | 経験者の視点・思考プロセス |
| 📌 | 操作前の確認事項 |
| ✅ | 完成チェックリスト（ハンズオン・混合） |
| 📖 | この Chapter の進め方（2層以上の場合） |

## 2. 構造

### Section の種類

| 種類 | 内容 |
|---|---|
| **概念** | 意義・仕組み・使い方を解説 |
| **ハンズオン** | 概念で学んだ機能を実践。冒頭に概念 Section への逆リンク |
| **混合** | 概念を学びながらすぐに手を動かす |

### 見出しの設計

**Why → What → How**（抽象→具体）の流れで組む。

### テンプレート

全 Section 共通の骨格に、種類ごとの差分を組み合わせる。

```markdown
# [セクション番号] セクションタイトル

<!-- 前提がある場合: 📝 **前提知識** 行を挿入 -->
<!-- 概念（グループ先頭）の場合: グループ全体像を挿入 -->
<!-- ハンズオンの場合: 📝 **このハンズオンで使う機能** 行を挿入 -->

## 🎯 このセクションで学ぶこと

- 学習目標1
- 学習目標2

[1文で Section の流れを示す]

---

## 導入: [課題・痛みから入る見出し]

### 🧠 先輩エンジニアの思考プロセス

> [実務の経験に基づいて自由に語る]

---

<!-- 本文: 種類ごとに異なる（「種類ごとの差分」を参照） -->

---

## ✨ まとめ

- ポイント1
- ポイント2

---

次のセクションでは、[次の内容への接続]。
```

**次セクションへの接続**:
- プレーンテキストで記述する（Markdown リンクやカギ括弧は使わない）
- **次セクションへの接続文** は、OUTLINE.md の次セクションのゴールに含まれるキーワードを漏れなく含める
- **Chapter 最終セクション**: Chapter の学びの振り返り + 次の Chapter への橋渡し
- **Part 最終セクション**: 上記に加え、Part 全体の振り返り + 次の Part への橋渡し
- **教材全体の最終セクション**: 教材全体の締め括り文を記述する（学んだことの振り返り + 次のステップへの示唆）

### 種類ごとの差分

**概念**:

- 本文は `## 見出し` で構成する
- 2層以上の場合、親グループの最初の Section のみ、タイトル直後にグループ全体像を含める。種類が混合でもグループ先頭であれば同様に含める。グループ全体像は `##` 見出しを使わず、タイトル直後に直接記述する。構成: ① 概要文（1-2文）② セクション一覧テーブル（セクション / テーマ / 種類）③ `📖 **この Chapter の進め方**: [学習の流れ]`

**ハンズオン**:

- タイトル直後に `📝 **このハンズオンで使う機能**: [機能名]（[セクション名] で学習）` を挿入
- 導入の後に `## 📌 [作業名]前の確認`（チェックリスト形式）を挿入
- 本文は `## 🏃 実践: [タイトル]` の中に `### 🏃 Step N: [タイトル]` の連番で構成する
- 本文の後、まとめの前に `## ✅ 完成チェックリスト` を挿入
- 長いコードは `<details>` で折りたたむ

**混合**:

- 本文の前半は `## 見出し`（概念解説）で構成する
- 本文の後半は `## 🏃 実践: [タイトル]` の中に `### 🏃 Step N: [タイトル]` の連番で構成する
- 本文の後、まとめの前に `## ✅ 完成チェックリスト` を挿入

**エラーフォーマットの使い分け**:
- エラーメッセージ（コードブロック）を伴う場合 → blockquote 形式（下記テンプレート）
- エラーメッセージなしの一般的な注意 → `⚠️ **注意**: ...` でインラインに記述

> ⚠️ **よくあるエラー**: [概要]
>
> ```
> [エラーメッセージ例]
> ```
>
> **原因**: [なぜ発生するか]
>
> **対処法**: [解決手順]

---

## 3. コンテンツ

### コードの見せ方

- コードを含む場合、コードブロック上部にファイルパスをコメントで明示する。コメント構文は言語タグに合わせる:
  - `java` / `typescript` / `javascript` / `php`: `// filepath`
  - `tsx` / `jsx`: `{/* filepath */}`
  - `yaml` / `properties` / `dockerfile` / `bash`: `# filepath`
  - `css`: `/* filepath */`
  - `xml`: `<!-- filepath -->`
  - `sql`: `-- filepath`
- コードブロックには必ず言語を指定する。非コードコンテンツ（ディレクトリツリー、コマンド出力等）には `text` を使用する
- diff 表示は使わない（テキストで説明し変更後のコードを示す）
- **参考資料のコードを引用する場合は正確性を最優先する**。コードを簡略化して掲載する場合は、コードブロックの直前に「以下は主要部分の抜粋です」と明記し、省略した重要な処理を本文で補足する。ファイルパス・変数名・設定値は実ファイルと完全に一致させる
- Blade テンプレートのコードは言語タグ `blade`（描画環境が未対応なら `html`）。Blade 内にファイルパスを書く場合は `{{-- filepath --}}` を使う。Mermaid 図は言語タグ `mermaid`

### 解説の深さ

- OUTLINE.md に前提（`前提:` フィールド）が記載されている場合、🎯 の前に `📝 **前提知識**: このセクションは○○の内容を前提としています。` と明記する
- 前提知識（概念）は概念の再入門を省略する。ただし Laravel での具体的な実現（記法・作法・差異）は省略せず解説する。既習との差異・組み合わせ時の注意点は特に丁寧に
- Laravel 固有の概念は初出時に丁寧に解説し、2回目以降は初出 Section への参照
- 同じ概念を別の箇所で繰り返さない
- 導入は「これがないと何が困るか」（課題）から入る
- 処理の流れ・サイクル・リレーション・ER 図は **Mermaid** で視覚化する（MkDocs Material / GitHub Pages で描画される）。コードブロックの言語タグは `mermaid`
- 未説明の用語をツリーやリストに含める場合は、簡潔なコメントか後続セクションへの参照を付ける
- つまずきポイントを先回りする（⚠️ で説明）
- 先の Section で扱う内容は「今は概要だけ把握すれば十分です」と明記する
- 画像（概念図）は執筆段階では挿入しない。**各概念 Section に 1 枚**を方針とし、後で `/illustrate` で生成・挿入する（挿入位置も /illustrate が計画するため、執筆時は TODO プレースホルダーも置かない）
- Section あたりの目安文字数: 最低 4,000 文字（概念 Section は Why → What → How を丁寧に書くと自然に超えることが多い。冗長を避けつつ概念理解を優先する）

### 実践パートの品質

🏃 実践・ハンズオンを含む Section では、以下を守る:

- **手順の再現性を保証する**: 各ステップを記載どおりに実行した場合に期待する結果が得られることを、実際の動作から確認する。特に Git 操作（`git checkout`, `git restore`, `git reset` 等）は、ステージ状態・ブランチ状態を正確に追跡し、クリーンアップ手順まで含める
- **前提状態を明示する**: 各ステップの実行前に必要な状態（コンテナ起動済み、特定ブランチにいる等）を `📌` チェックリストで示す
- **副作用のクリーンアップ**: 実践で作成したブランチ・コミット・ファイル変更は、セクション末尾で元に戻す手順を必ず含める

### コピペで完成するハンズオンの留意点

複数 Section をまたいで 1 つの成果物を積み上げるハンズオン（総合ハンズオン等）では、「学習者が最初の Section から順に、各コマンド・コードをそのままターミナル / エディタにコピペして進めると成果物が完成する」ことを保証する。各 Section の境界でビルド / 実行が通ることを前提に、次を守る。

**コードはコピペでビルド / 実行が通る形にする**

- 定型コード（getter / setter 等）を「エディタで生成」に委ねず、後続 Section がそれを参照するなら省略せず全掲載する。揃っていないとビルドが通らない（生成方法は補足に留める）
- **前方参照しない**: 各 Step のコードは、その時点で既に定義済みのものだけを参照する。動作確認より後の Step で追加する処理を先に呼ばない
- 「既存ファイルに追加 / 置換」する Step では、**どのファイルのどこへ** 入れるか（先頭・特定ブロック内・全体差し替え等）を明示し、**新規に必要な依存だけ** を示す（既に取り込み済みのものを再掲して混乱させない）

**足場（環境・配置）の手順を省略しない**

- ディレクトリ作成・展開・`cd`・エディタで開く操作・ファイルの配置先を省略しない。各コマンドの **実行場所（カレントディレクトリ）を毎回曖昧にしない**
- OS 差を補足する（コマンド名・パス区切り等）。初出で一度示し「以降も同様」と添える

**動作確認は実際の挙動と一致させる**

- レスポンス本文がない成功（HTTP 201 等）はステータスを可視化する手段（`curl -i` 等）を示す
- 出力例（ソート結果・整形結果・ログ）は、掲載したコードを実行したときの **実際の結果** と一致させる。憶測で書かない
- 想定どおり起きる中間状態（認証必須パスで 401 になる等）は「正常です」と明記し、非同期処理は結果の確認方法まで示す

**フレームワークの挙動は裏取りしてから断定する**

- 「このコードで動く」「この設定で解決する」が成否を分ける箇所（依存名・読み込まれる設定・アノテーションや記法の要否・バージョン専用機能等）は、公式ドキュメント等で **対象バージョンでの挙動** を確認してから書く

**コードを書かない設計 Section の表現**

- 設計だけで成果物（ファイル）が生まれない Section では、「完成チェックリスト」「成果物が揃っている」といった表現を避け、「設計を説明できる / 決められた」など **理解・意思決定** として書く。冒頭の `📝` も「使う機能」ではなく「活かす概念」等、実態に合わせる
