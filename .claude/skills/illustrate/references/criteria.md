# 中心概念とタイプの選定ガイド

教材の Section に概念図を入れるとき、「どの Section に入れるか」「その Section の何を・どのタイプで描くか」を判断するための基準。

## 0. 画像の密度方針

**本教材の方針: [A] 各概念 Section に 1 枚。** 概念主軸の教材のため、スコープ内の「概念」種別 Section すべてに、導入 🧠 直後へ 1 枚ずつ入れる。判断は「入れるか否か」ではなく「何を・どのタイプで描くか」。執筆段階では画像を挿入せず、`/illustrate` 実行時にまとめて生成・挿入する。

- **[A] 各概念 Section に 1 枚**: スコープ内の「概念」種別 Section すべてに、導入 🧠 直後へ 1 枚ずつ入れる。判断は「入れるか否か」ではなく「何を・どのタイプで描くか」になる。図が充実するがコストは高め
- **[B] 判断ベース（デフォルト）**: 下の観点に強く該当する箇所だけに入れる（目安 1 Chapter あたり 1〜2 枚）。図は少なめ・低コスト

方針 [A] でも、各 Section の「中心概念の選定」「タイプの選定」は以下の手順に従う。方針 [B] では、まず「入れる価値があるか」を観点で判断してから同じ手順に進む。

## 1. 入れる価値があるか（方針 [B] の判断観点）

以下に強く該当するほど、AI 画像の効果が高い。

| 観点 | 説明 |
|------|------|
| **メンタルモデルの可視化** | 抽象的な概念を直感的なイメージで掴ませたい |
| **鳥瞰** | 複雑なシステムの全体像を「地図」として俯瞰させたい |
| **Before/After の対比** | 技術導入の「なぜ」を課題→解決で実感させたい |
| **比喩・アナロジー** | 🧠 で使った比喩を絵にして語りと響かせたい |
| **パラダイムシフト** | 読者の思考の転換点を 1 枚で示したい |

### Mermaid との使い分け

- **Mermaid = How it works**（仕組みの正確な図解・処理順序・データの流れ）→ 本文中に配置。頻繁に更新されるものもこちら
- **AI 画像 = How to think about it**（概念の直感的な掴み）→ 導入 🧠 直後に配置

本文に Mermaid がある概念は、その再描画にせず、別角度（比喩・鳥瞰・Before/After）から描く。

## 2. 中心概念を 1 つ選ぶ

画像は導入の 🧠（[人格名]はこう考える）の直後に置かれる。読者が Section を開いて最初に出会う図なので、**この Section で一番つかんでほしい 1 つのメンタルモデル** を選ぶ。

- **サマリー図として描く**: 要点を一目で伝える役割。構図は `style-guide.md` のテンプレートに従って具体的に指定する
- **🧠 の語りに呼応させる**: 導入の 🧠 が使っている比喩・痛み・たとえ話をそのまま絵にするのが最も効果的（画像が 🧠 の直後に並ぶため、語りと図が響き合う）
- **1 枚 1 概念**: 欲張らない。Section の主役概念だけを描く（要素は 5〜8 個以内）

## 3. 構図を選ぶ（3 つの型）

中心概念が決まったら、どう描くか（構図）を選ぶ。次の 3 つは **構図の引き出しであって、排他的な分類ではない**。1 枚が複数を兼ねてよい（例: 比喩を使った対比図）。

| # | 構図 | どう描くか |
|---|------|------|
| 1 | **比喩図（メンタルモデル）** | 抽象概念を日常の物・場面に置き換え、対応関係をラベルで併記する |
| 2 | **対比図（Before/After）** | 左右に分割し、旧→新・課題→解決・思考の転換を対比する |
| 3 | **鳥瞰図（地図）** | 全体を街・施設の地図に見立て、各要素を日常の物に擬物化して一望させる |

各構図の描き方（配色・レイアウト）は `style-guide.md` の「画像の種類別ガイド」を参照する。

## 4. 描きにくい Section の扱い

一見「絵にしにくい」Section もある。その場合でも、次のいずれかの角度が見つかることが多い:

- **導入の課題から Before/After**: 導入は「これがないと何が困るか」から始まる（writing.md）。その課題→解決を対比で描く
- **既習との対比そのものを図に**: 読者が既に知っている概念との対応・差分を比喩や対比で描く
- **Section の鳥瞰**: その Section で学ぶ要素を地図的に並べ、「これから何を学ぶか」を一目で示す

それでも意味のある図が立たない場合は、無理に飾り画像を作らず、plan の報告でその旨を述べてユーザーの判断を仰ぐ。

## plan の報告フォーマット

```markdown
### [Section 番号] [Section タイトル]

- **状態**: 未生成 / 生成済み
- **中心概念**: [図にする 1 つの概念]
- **タイプ**: [3 つの構図のどれか]
- **画像名**: `<section番号>-<concept-slug>`
- **プロンプト概要**: [画像に描く内容の概要]
```
