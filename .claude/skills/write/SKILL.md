# 教材の執筆ワークフロー

**入力**: $ARGUMENTS

## スコープの解釈

本教材は 3層（Part > Chapter > Section）。スコープは次の 3 パターンで指定する（番号のみでも可）。

| 入力 | 対象 |
|---|---|
| `Part 2`（または `2`） | Part 全体 |
| `Chapter 2-1`（または `2-1`） | Chapter 全体 |
| `Section 2-1-3`（または `2-1-3`） | Section 単体 |

`全て` / `残り全部` は全 Section が対象（`curriculums/` に既存ファイルがある Section はスキップ）。

ディレクトリパス・タイトルは CLAUDE.md の命名規則と OUTLINE.md に従う。

## 1. 準備

1. `CLAUDE.md` を Read し、哲学（ペルソナ・コンセプト・ゴール）と階層構造を確認
2. `OUTLINE.md` を Read し、対象スコープの設計を把握（ゴール、種類、参考資料、前提 Section 等）
3. 参考資料を取得する。OUTLINE.md の各 Section に記載された参考資料を確認し、**ローカルファイルパスは Read ツールで読み込む**。URL がある場合は WebFetch、必要に応じて WebSearch でリサーチする。取得した情報から教材に含める数値・仕様・コードパターンを箇条書きで整理し、執筆時は記憶ではなくこの整理結果を参照する。複数 Section をまとめて書く場合は、参考資料を**事前に一括取得して整理結果を1つにまとめ**（重複取得を防ぐ）、Section 横断で統一すべき事実（名前パターン・数値・用語）も含める
4. `curriculums/` に既存ファイルがあればサブエージェントで分析（用語の一貫性、前後との接続）
5. 実践プロジェクトがある場合: Section 間の依存関係（前の Section で作成したコード・設定を後の Section が前提とする箇所）を確認し、整合性を保って執筆する

## 2. 方針合わせ

執筆に入る前に、見出し構成をユーザーに提示して方向性を揃える。Part / Chapter 単位のときは Chapter ごとにまとめる。

- **構成判断**: 対象スコープの各 Section について、見出しを Why → What → How の流れで箇条書きに列挙して提示する（1行サマリーで済ませず、実際の見出し単位で出す）。OUTLINE.md の設計から外れる場合は理由を添える
- **接続の確認**: 最終 Section の接続先（次 Section / 次 Chapter / 次 Part / 教材の締め括り）を明示的に確認する
- **不明点**: 判断に迷う点があれば質問する

ユーザーの反応を受けてから執筆に入る。

## 3. 執筆

CLAUDE.md の哲学を踏まえ、`.claude/rules/writing.md` のルール・テンプレートに沿って書く。Part / Chapter 単位など複数 Section を含む場合は、含まれる全 Section を順に執筆する（サブエージェントへの分担も可）。

**サブエージェントに分担させる場合の情報の渡し方**:

- **ファイルパスで渡す**: ルールファイル（`.claude/rules/writing.md`）、OUTLINE.md、参考資料のローカルファイルはパスを渡してサブエージェント自身に Read させる。プロンプトへのルール要約やコード全文の埋め込みは避ける
- **記憶からの再構成を禁止する**: 参考コードは必ず Read 結果からコピーする。メインエージェントの要約を経由させない
- **数値はソースで検証させる**: 個数・件数等の数値を記載する場合、参考資料の整理結果をそのまま使わず、実ファイルを Read してカウントを検証するよう指示する
- **セクションタイトルは正確に引用させる**: 前提知識や接続文でセクション名を参照する際は OUTLINE.md のタイトルを正確に使い、要約・言い換えをしないよう指示する

## 4. セルフチェック・完了

以下を検証し、問題があれば修正する:

1. **機械的検証**（Grep ツールで実行）:
   - `**太字**` 直後のスペース欠落（全角括弧 `（` を含む）
   - 言語指定なしのコードブロック（bare ` ``` `）
   - ダッシュ記号（`——`、`—`、`–`）の使用
2. **OUTLINE との整合**:
   - 各 Section の末尾接続文が、OUTLINE.md の次 Section のゴールキーワードを網羅しているか
   - OUTLINE の `前提:` フィールドが `📝 **前提知識**` として反映されているか
   - 本文が OUTLINE のゴールの範囲内に収まっているか（ゴールにないトピックを追加していないか）
3. **コードの正確性**（実践プロジェクトを扱う場合）:
   - ファイルパスが実在すること
   - クラス名・メソッド名・変数名が実ファイルと一致すること
   - 簡略化したコードに「抜粋」注記があること
   - 数値（個数・件数）が実ファイルと一致すること
4. 参考資料との照合: 教材に記載した数値・仕様が準備で整理した内容と一致するか確認
5. セルフチェック結果を報告し、`/review` の実行を提案する
