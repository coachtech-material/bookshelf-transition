# 4-3 ハンズオン: 投稿の所有者だけ編集・削除できるようにする

📝 **このハンズオンで使う機能**: Policy（4-2 Policy の作成・登録・適用 で学習）、`authorize()` と `@can`（4-1 認可とは／Gate・4-2 Policy の作成・登録・適用 で学習）

📝 **前提知識**: このセクションは 4-2 Policy の作成・登録・適用 の内容を前提としています。

## 🎯 このセクションで学ぶこと

- スターターキットをクローンして起動し、認可が未実装の状態を確認する
- `PostPolicy` を作成・登録し、所有者だけが投稿を編集・削除できるようにする
- 非所有者には 403 を返し、`@can` で編集・削除ボタンを出し分ける

このセクションでは、認証・モデル・コントローラ・Blade がそろい認可だけが空いているスターターキットに、4-1・4-2 で学んだ認可を実装します。

---

## 導入: 認可がないと、誰でも他人の投稿を触れる

このハンズオンで使うスターターキットは、Fortify による認証・`Post` モデル・`PostController`・シーダー・Blade まで用意されています。投稿一覧と編集・更新・削除のルートもあります。足りないのは認可だけです。

そのため、今のままでは **ログインしていれば誰でも、他人の投稿に編集・削除ボタンが表示され、実際に編集・削除できてしまいます**。これが塞ぐべき穴です。`PostPolicy` を実装し、「編集・削除してよいのは投稿した本人だけ」という制御を加えていきます。

### 🧠 先輩エンジニアの思考プロセス

> 認可のレビューで一番効くのは、自分で「他人のデータを触れてしまう」のを一度体験することです。ボタンを消して満足せず、URL を直打ちして 403 になるところまで確かめる。この確認の癖が、画面では気づけない認可漏れを防いでくれます。

## 🏃 実践: スターターキットに所有者認可を実装する

### 🏃 Step 1: スターターキットをクローンする

プロジェクトを置きたいディレクトリ（例: ホームディレクトリ）で、リポジトリをクローンして中に入ります。

```bash
# プロジェクトを置きたいディレクトリで実行（例: cd ~ してから）
git clone https://github.com/coachtech-material/laravel-authorization-starter.git
cd laravel-authorization-starter
```

### 🏃 Step 2: 環境を準備して起動する

まず `.env` を用意します。スターターキットには Sail 向けの設定が入った `.env.example` が含まれています。

```bash
# laravel-authorization-starter ディレクトリで実行
cp .env.example .env
```

クローン直後は `vendor` がなく `sail` コマンドがまだ使えません。2-3 と同じく、PHP と Composer の入った Docker イメージを一時的に借りて、依存パッケージをインストールします。

```bash
# laravel-authorization-starter ディレクトリで実行
docker run --rm \
    -u "$(id -u):$(id -g)" \
    -v "$(pwd):/var/www/html" \
    -w /var/www/html \
    -e COMPOSER_CACHE_DIR=/tmp/composer_cache \
    laravelsail/php82-composer:latest \
    composer install
```

`vendor` ができたので、ここからは `sail` が使えます。コンテナを起動し、アプリケーションキーの生成と、マイグレーション・シーディングを行います。

```bash
# laravel-authorization-starter ディレクトリで実行
sail up -d
sail artisan key:generate
sail artisan migrate --seed
```

`migrate --seed` で、テスト用のユーザー 2 人（`usera@example.com` と `userb@example.com`、どちらもパスワードは `password`）と、各ユーザーの投稿 2 件ずつ（合計 4 件）が作成されます。

> ⚠️ **注意**: 初回の `sail up` は、コンテナのイメージ構築のため数分かかることがあります。完了するまで待ってください。

> ⚠️ **よくあるエラー**: `sail up -d` の直後に `migrate` を実行すると、MySQL の初期化が終わっておらず次のエラーが出ることがあります。
>
> ```text
> SQLSTATE[HY000] [2002] Connection refused
> ```
>
> **原因**: MySQL コンテナがまだ起動準備中で、接続を受け付けていない。
>
> **対処法**: `sail ps` で `mysql` が `running` になっている（初回は数十秒かかります）ことを確認してから、`sail artisan migrate --seed` を実行し直してください。

### 🏃 Step 3: ログインして現状を確認する

ブラウザで `http://localhost/login` を開き、`usera@example.com` / `password` でログインします。ログイン後、投稿一覧 `http://localhost/posts` を開いてください。

ここで、認可がない状態を確認します。

- ユーザー A・B の投稿が 4 件すべて表示される
- **ユーザー B の投稿にも、編集・削除ボタンが表示される**
- ユーザー B の投稿の「編集」を開くと、編集フォームが開き、更新できてしまう

これが、これから塞ぐ問題です。今はログインしているだけで、他人の投稿を操作できてしまいます。

### 🏃 Step 4: PostPolicy を作る

`Post` モデルに紐づく Policy を生成します。

```bash
# laravel-authorization-starter ディレクトリで実行
sail artisan make:policy PostPolicy --model=Post
```

`app/Policies/PostPolicy.php` が生成されます。`viewAny` / `view` / `create` / `update` / `delete` などのメソッドが、中身が空の状態で並んでいます。

### 🏃 Step 5: 所有者チェックを書く

`app/Policies/PostPolicy.php` を開き、内容を次のように **まるごと書き換えます**。生成時に並んでいた `viewAny` などのメソッドは今回使わないので、所有者チェックを書く `update` と `delete` だけを残します。判定は「投稿の `user_id` がログインユーザーの `id` と一致するか」です。

```php
<?php
// app/Policies/PostPolicy.php

namespace App\Policies;

use App\Models\Post;
use App\Models\User;

class PostPolicy
{
    public function update(User $user, Post $post): bool
    {
        return $user->id === $post->user_id;
    }

    public function delete(User $user, Post $post): bool
    {
        return $user->id === $post->user_id;
    }
}
```

### 🏃 Step 6: AuthServiceProvider に登録する

`app/Providers/AuthServiceProvider.php` を開き、内容を次のように **まるごと書き換えます** （`use` を 2 つ加え、`$policies` 配列に `Post` と `PostPolicy` の対応を登録しています）。

```php
<?php
// app/Providers/AuthServiceProvider.php

namespace App\Providers;

use App\Models\Post;
use App\Policies\PostPolicy;
use Illuminate\Foundation\Support\Providers\AuthServiceProvider as ServiceProvider;

class AuthServiceProvider extends ServiceProvider
{
    /**
     * The model to policy mappings for the application.
     *
     * @var array<class-string, class-string>
     */
    protected $policies = [
        Post::class => PostPolicy::class,
    ];

    /**
     * Register any authentication / authorization services.
     */
    public function boot(): void
    {
        //
    }
}
```

### 🏃 Step 7: コントローラに authorize を追加する

`app/Http/Controllers/PostController.php` を開きます。`edit` / `update` / `destroy` の各メソッドには、`// TODO: $this->authorize('update', $post); を追加する`（`destroy` のみ `'delete'`）というコメントが置かれています。このコメント行を、実際の `authorize` 呼び出しに置き換えます（`edit` と `update` は `update` の認可、`destroy` は `delete` の認可）。`index` は変更しません。

ファイル全体は次のようになります。このまま貼り替えても構いません。

<details>
<summary>app/Http/Controllers/PostController.php（全体）</summary>

```php
<?php
// app/Http/Controllers/PostController.php

namespace App\Http\Controllers;

use App\Models\Post;
use Illuminate\Http\Request;

class PostController extends Controller
{
    public function index()
    {
        $posts = Post::with('user')->latest()->get();

        return view('posts.index', compact('posts'));
    }

    public function edit(Post $post)
    {
        $this->authorize('update', $post);

        return view('posts.edit', compact('post'));
    }

    public function update(Request $request, Post $post)
    {
        $this->authorize('update', $post);

        $validated = $request->validate([
            'title' => 'required|string|max:255',
            'content' => 'required|string',
        ]);

        $post->update($validated);

        return redirect()->route('posts.index');
    }

    public function destroy(Post $post)
    {
        $this->authorize('delete', $post);

        $post->delete();

        return redirect()->route('posts.index');
    }
}
```

</details>

これで、他人の投稿の編集・更新・削除を試みると、`authorize` が 403 を返すようになります。

### 🏃 Step 8: ビューに @can を追加する

最後に、一覧画面で「他人の投稿には編集・削除ボタンを出さない」ようにします。`resources/views/posts/index.blade.php` を開くと、各投稿カードの中に `{{-- TODO: @canディレクティブを追加して、投稿者のみに編集・削除ボタンを表示する --}}` というコメントと、その下に編集リンク・削除フォームがあります。この `actions` ブロックを、`@can` で囲んだ形に置き換えます。

```blade
{{-- resources/views/posts/index.blade.php （actions ブロックを置き換える） --}}
<div class="actions">
    @can('update', $post)
        <a href="{{ route('posts.edit', $post) }}" class="btn-edit">編集</a>
    @endcan
    @can('delete', $post)
        <form action="{{ route('posts.destroy', $post) }}" method="POST" style="display: inline;">
            @csrf
            @method('DELETE')
            <button type="submit" class="btn-delete">削除</button>
        </form>
    @endcan
</div>
```

編集ボタンは `update`、削除ボタンは `delete` の認可で囲みました。どちらも `PostPolicy` の判定を使うため、所有者にだけ表示されます。

### 🏃 Step 9: 認可が効くことを検証する

ユーザー A（`usera@example.com`）でログインしたまま、`http://localhost/posts` を再読み込みします。

- ユーザー A 自身の投稿には、編集・削除ボタンが表示される
- **ユーザー B の投稿には、編集・削除ボタンが表示されない**

ボタンが消えるのは表示の制御です。処理の制御も効いているかを、URL の直打ちで確かめます。シーダーは先にユーザー A の投稿 2 件（id 1・2）、続いてユーザー B の投稿 2 件（id 3・4）を作るので、ユーザー B の投稿は id 3・4 です。ユーザー A でログインしたまま、ユーザー B の投稿の編集 URL に直接アクセスします。

```text
http://localhost/posts/3/edit
```

ボタンを消していても、`authorize` が働いて **403 Forbidden** が返れば成功です。これで、画面（`@can`）と処理（`authorize`）の両方で、所有者だけが編集・削除できる状態になりました。

💡 念のため、`userb@example.com` / `password` でログインし直すと、今度はユーザー B の投稿（id 3・4）に編集・削除ボタンが表示され、ユーザー A の投稿には表示されないことも確認できます。

### 🏃 Step 10: 後片付けをする

このプロジェクトは、2-3 のサンドボックスとは独立した使い捨てのクローンです。確認が終わったら、コンテナを停止します。

```bash
# laravel-authorization-starter ディレクトリで実行
sail down
```

今後使わないなら、停止したあとディレクトリごと削除して構いません。また、この先サンドボックス（2-3）を使うときは、ポート競合を避けるため、こちらを `sail down` で止めてから起動してください。

## ✨ まとめ

- 認証・モデル・コントローラ・Blade がそろったスターターキットに、認可だけを実装した
- `make:policy --model` で Policy を生成し、`update` / `delete` に所有者チェック（`$user->id === $post->user_id`）を書いた
- `AuthServiceProvider` の `$policies` に登録し、コントローラで `authorize()`、ビューで `@can` を適用した
- ボタンを隠す `@can`（表示の制御）と、403 を返す `authorize()`（処理の制御）は、両方そろって安全になる。URL 直打ちでの 403 まで確認した

この Chapter では、認可を扱いました。認証（誰か）と認可（何をしてよいか）の違いを整理し、Gate による単純な認可を確認したうえで、Policy を作成・登録・適用し、所有者だけが編集・削除できる制御をスターターキットに実装しました。非所有者には 403 を返し、`@can` でボタンを出し分けるところまで、手を動かして確かめられました。

次の Chapter では、集計とパフォーマンスに進みます。「カテゴリごとのタスク数」「タスクが多い順のランキング」のような、関連レコードを数えたり平均したりする処理を扱います。`withCount` / `withAvg` / `withSum` で関連を集計し、`has` / `whereHas` で関連の有無や条件で絞り込み、`orderByDesc` で集計結果を並べ替えてランキングを作ります。これらを、2-3 のサンドボックスの Tinker で実際に動かして確認します。
