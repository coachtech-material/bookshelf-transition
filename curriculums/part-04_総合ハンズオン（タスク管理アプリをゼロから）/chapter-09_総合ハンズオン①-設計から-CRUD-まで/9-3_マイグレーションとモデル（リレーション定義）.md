# 9-3 マイグレーションとモデル（リレーション定義）

📝 **このハンズオンで使う機能**: ピボットテーブルの設計（3-1 多対多とピボットテーブル で学習）、`belongsToMany`（3-1 多対多とピボットテーブル で学習）

📝 **前提知識**: このセクションは 3-1 多対多とピボットテーブル と 9-2 環境構築と提供 Blade アセットの配置 の内容を前提としています。

## 🎯 このセクションで学ぶこと

- 9-1 で設計した 6 つのテーブルのマイグレーションを作り、外部キーと ON DELETE CASCADE を設定する
- モデルに 1 対多・多対多のリレーションと `$fillable` / `$casts` を定義する
- ファクトリとシーダーを用意し、`migrate:fresh --seed` で初期データを投入する

このセクションでは、9-1 の設計図をマイグレーションとモデルに落とし込み、初期データを入れて、データベースの土台を完成させます。

---

## 導入: 設計図を、動くテーブルとモデルにする

9-1 で、`users`・`categories`・`tags`・`tasks` の 4 実体と、`task_tag` のピボットを設計しました。9-2 で画面とルートを置きました。残るは、その画面が読み書きするデータの器、つまりテーブルとモデルです。

ここでは、外部キーとカスケード削除を正しく設定し、多対多のリレーションを `belongsToMany` で定義します。タスクとタグの多対多は、3-1 で学んだ 2 パターンのうち純粋なピボット（`task_tag`）で実装します。最後にシーダーで初期データを入れ、画面で確認できる状態にします。

### 🧠 先輩エンジニアの思考プロセス

> マイグレーションで一番事故りやすいのが、外部キーの順序とカスケードです。`tasks` が `categories` を参照するなら、`categories` のマイグレーションが先に走らないと外部キーが張れません。実行順は **ファイル名の日時順** で決まります。やっかいなのがピボットの `task_tag` で、`tasks` を参照するのにファイル名が `tasks` より前に並びやすい（`task_tag` < `tasks`）。同じ秒に生成されるとここで事故るので、生成後に時刻を遅らせて確実に最後へ回すようにしています。カスケードは「親を消したら子も消す」を意図どおりに設定できているか、シーダーを入れた後に一度試して確かめます。

## 🏃 実践: テーブルとモデルを作る

### 🏃 Step 1: モデルとマイグレーションを生成する

モデルとマイグレーション（一部はファクトリも）を生成します。`-m` はマイグレーション、`-f` はファクトリを同時に作るオプションです。**参照される側（カテゴリ・タグ）から先に** 作り、続いて `tasks`、最後にピボット（`task_tag`）を作ります。

```bash
# task-app ディレクトリで実行
sail artisan make:model Category -mf
sail artisan make:model Tag -mf
sail artisan make:model Task -mf
sail artisan make:migration create_task_tag_table
```

これで、`Category` / `Tag` / `Task` のモデル・マイグレーション・ファクトリと、`task_tag` のマイグレーションが生成されます（`task_tag` はピボットなので、モデルは作りません）。

🔑 マイグレーションは、生成した順ではなく **ファイル名の日時順** に実行されます。ピボットの `task_tag` は `tasks` を参照するので、必ず `tasks` より後に実行される必要があります。ところが `make:model Task` と `make:migration create_task_tag_table` が同じ秒に生成されると、`create_task_tag_table` は `create_tasks_table` より文字列順で前に来てしまい（11 文字目が `_` < `s`）、`task_tag` が `tasks` より先に走って外部キーエラーになります。これを防ぐため、ピボットのファイル名の時刻部分を遅らせて、確実に最後に実行されるようにします。

```bash
# task-app ディレクトリで実行（task_tag を必ず最後に実行させる）
PIVOT_DATE=$(basename database/migrations/*_create_tasks_table.php | cut -c1-10)
mv database/migrations/*_create_task_tag_table.php \
   "database/migrations/${PIVOT_DATE}_999999_create_task_tag_table.php"
```

`tasks` マイグレーションと同じ日付で、時刻部分を `999999`（実在しない値ですが、ファイル名は文字列順で並ぶため「必ず最後」を保証できます）に付け替えています。これで実行順が `categories` → `tags` → `tasks` → `task_tag` に固定されます。`ls database/migrations` で、`create_task_tag_table` のファイルが一番下（最新の日時）に来ていれば成功です。

### 🏃 Step 2: マイグレーションを編集する

生成された 4 つのマイグレーションファイル（`database/migrations/` 配下）を、順に編集します。ファイル名の日時部分（`xxxx_xx_xx_xxxxxx`）は生成時刻によって変わるので、`create_xxx_table` の部分で見分けてください。

まず `categories` テーブル。`name` に一意制約を付けます。

```php
<?php
// database/migrations/xxxx_xx_xx_xxxxxx_create_categories_table.php

use Illuminate\Database\Migrations\Migration;
use Illuminate\Database\Schema\Blueprint;
use Illuminate\Support\Facades\Schema;

return new class extends Migration
{
    public function up(): void
    {
        Schema::create('categories', function (Blueprint $table) {
            $table->id();
            $table->string('name')->unique();
            $table->timestamps();
        });
    }

    public function down(): void
    {
        Schema::dropIfExists('categories');
    }
};
```

`tags` テーブルも同じ形です。

```php
<?php
// database/migrations/xxxx_xx_xx_xxxxxx_create_tags_table.php

use Illuminate\Database\Migrations\Migration;
use Illuminate\Database\Schema\Blueprint;
use Illuminate\Support\Facades\Schema;

return new class extends Migration
{
    public function up(): void
    {
        Schema::create('tags', function (Blueprint $table) {
            $table->id();
            $table->string('name')->unique();
            $table->timestamps();
        });
    }

    public function down(): void
    {
        Schema::dropIfExists('tags');
    }
};
```

`tasks` テーブルは、所有者（`user_id`）と分類（`category_id`）への外部キーを持ちます。`user_id` はカスケード削除（ユーザーを消したらそのタスクも消す）、`category_id` は外部キーの既定の動作（参照されているカテゴリは削除できない）にします。

```php
<?php
// database/migrations/xxxx_xx_xx_xxxxxx_create_tasks_table.php

use Illuminate\Database\Migrations\Migration;
use Illuminate\Database\Schema\Blueprint;
use Illuminate\Support\Facades\Schema;

return new class extends Migration
{
    public function up(): void
    {
        Schema::create('tasks', function (Blueprint $table) {
            $table->id();
            $table->foreignId('user_id')->constrained()->onDelete('cascade');
            $table->foreignId('category_id')->constrained();
            $table->string('title');
            $table->text('description')->nullable();
            $table->string('status')->default('pending');
            $table->date('due_date')->nullable();
            $table->timestamps();
        });
    }

    public function down(): void
    {
        Schema::dropIfExists('tasks');
    }
};
```

🔑 `category_id` の `->constrained()` には `onDelete` を付けていません。外部キーの既定の動作では、参照されているカテゴリ（タスクが紐づくカテゴリ）はデータベースのレベルでも削除できません。その手前で、アプリ側のガード（9-5）が「タスクが紐づくカテゴリは削除できません」と分かりやすく止めます。データベース制約とアプリのガードの二段構えで、整合性を守ります。

`task_tag` テーブルは、3-1 で学んだ **純粋なピボット** です。`task_id` と `tag_id` の複合主キーを持ち、`id` も `timestamps` も置きません。両方の外部キーをカスケード削除にします。

```php
<?php
// database/migrations/xxxx_xx_xx_xxxxxx_create_task_tag_table.php

use Illuminate\Database\Migrations\Migration;
use Illuminate\Database\Schema\Blueprint;
use Illuminate\Support\Facades\Schema;

return new class extends Migration
{
    public function up(): void
    {
        Schema::create('task_tag', function (Blueprint $table) {
            $table->foreignId('task_id')->constrained()->onDelete('cascade');
            $table->foreignId('tag_id')->constrained()->onDelete('cascade');
            $table->primary(['task_id', 'tag_id']);
        });
    }

    public function down(): void
    {
        Schema::dropIfExists('task_tag');
    }
};
```

### 🏃 Step 3: モデルにリレーションを定義する

4 つのモデルに、リレーションと `$fillable` を定義します。まず `app/Models/Category.php` を **まるごと書き換えます**。

```php
<?php
// app/Models/Category.php

namespace App\Models;

use Illuminate\Database\Eloquent\Factories\HasFactory;
use Illuminate\Database\Eloquent\Model;
use Illuminate\Database\Eloquent\Relations\HasMany;

class Category extends Model
{
    use HasFactory;

    protected $fillable = ['name'];

    public function tasks(): HasMany
    {
        return $this->hasMany(Task::class);
    }
}
```

`app/Models/Tag.php` を **まるごと書き換えます**。タグからタスクへは、`task_tag` を介した多対多です。

```php
<?php
// app/Models/Tag.php

namespace App\Models;

use Illuminate\Database\Eloquent\Factories\HasFactory;
use Illuminate\Database\Eloquent\Model;
use Illuminate\Database\Eloquent\Relations\BelongsToMany;

class Tag extends Model
{
    use HasFactory;

    protected $fillable = ['name'];

    public function tasks(): BelongsToMany
    {
        return $this->belongsToMany(Task::class, 'task_tag');
    }
}
```

`app/Models/Task.php` を **まるごと書き換えます**。タスクは、所有者（`user`）と分類（`category`）への `belongsTo`、タグへの `belongsToMany`（`task_tag`）を持ちます。`due_date` は日付としてキャストします。

```php
<?php
// app/Models/Task.php

namespace App\Models;

use Illuminate\Database\Eloquent\Factories\HasFactory;
use Illuminate\Database\Eloquent\Model;
use Illuminate\Database\Eloquent\Relations\BelongsTo;
use Illuminate\Database\Eloquent\Relations\BelongsToMany;

class Task extends Model
{
    use HasFactory;

    protected $fillable = [
        'user_id',
        'category_id',
        'title',
        'description',
        'status',
        'due_date',
    ];

    protected $casts = [
        'due_date' => 'date',
    ];

    public function user(): BelongsTo
    {
        return $this->belongsTo(User::class);
    }

    public function category(): BelongsTo
    {
        return $this->belongsTo(Category::class);
    }

    public function tags(): BelongsToMany
    {
        return $this->belongsToMany(Tag::class, 'task_tag');
    }
}
```

🔑 `due_date` を `$casts` で `date` にすると、データベースの文字列が Carbon の日付オブジェクトとして扱え、Blade で `$task->due_date?->format('Y/m/d')` のように整形できます。2-2 で確認したとおり、Laravel 10 のキャストは `casts()` メソッドではなく `$casts` プロパティで書きます。

最後に `app/Models/User.php` を **まるごと書き換えます**。標準の `User` に、所有するタスク（`tasks`、`hasMany`）を加えます。

```php
<?php
// app/Models/User.php

namespace App\Models;

use Illuminate\Database\Eloquent\Factories\HasFactory;
use Illuminate\Database\Eloquent\Relations\HasMany;
use Illuminate\Foundation\Auth\User as Authenticatable;
use Illuminate\Notifications\Notifiable;

class User extends Authenticatable
{
    use HasFactory, Notifiable;

    protected $fillable = [
        'name',
        'email',
        'password',
    ];

    protected $hidden = [
        'password',
        'remember_token',
    ];

    protected $casts = [
        'email_verified_at' => 'datetime',
        'password' => 'hashed',
    ];

    public function tasks(): HasMany
    {
        return $this->hasMany(Task::class);
    }
}
```

### 🏃 Step 6: マイグレーションとシーディングを実行する

テーブルを作り直し、初期データを投入します。

```bash
# task-app ディレクトリで実行
sail artisan migrate:fresh --seed
```

`migrate:fresh` は全テーブルを削除してからマイグレーションをやり直し、`--seed` で `DatabaseSeeder` を実行します。出力に、`categories`・`tags`・`tasks`・`task_tag` の作成と、`Database\Seeders\DatabaseSeeder` の実行が並べば成功です。

> ⚠️ **よくあるエラー**: `SQLSTATE[HY000]: General error: 1215 Cannot add foreign key constraint`（MySQL 8 では `1824 Failed to open the referenced table 'tasks'` と出ることもあります）
>
> **原因**: 参照される側のテーブルより先に、参照する側のマイグレーションが走っている。とくに `task_tag` は、`tasks` とファイル名の日時が同じ秒になると `tasks` より前に並び（`create_task_tag_table` < `create_tasks_table`）、`tasks` が無い状態で実行されて失敗します。
>
> **対処法**: Step 1 の最後で `task_tag` のファイル名を `..._999999_...` に付け替えていれば、`task_tag` は必ず最後に実行されるので、このエラーは起きません。すでにこのエラーが出た場合は、Step 1 の付け替えコマンドを実行してから、もう一度 `sail artisan migrate:fresh --seed` を実行してください。


次の文を 1 行ずつ実行します（チェーンは 1 行で書きます）。日本語はバイト列で表示されることがあるため、件数や ID で確認します。

```php
\App\Models\Task::first()->tags()->count();
```
