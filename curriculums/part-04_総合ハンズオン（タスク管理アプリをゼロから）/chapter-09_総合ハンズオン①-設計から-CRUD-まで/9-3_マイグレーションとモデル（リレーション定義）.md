# 9-3 マイグレーションとモデル（リレーション定義）

📝 **このハンズオンで使う機能**: ピボットテーブルの設計（3-1 多対多とピボットテーブル で学習）、`belongsToMany`（3-1 多対多とピボットテーブル で学習）

📝 **前提知識**: このセクションは 3-1 多対多とピボットテーブル と 9-2 環境構築と提供 Blade アセットの配置 の内容を前提としています。

## 🎯 このセクションで学ぶこと

- 9-1 で設計した 6 つのテーブルのマイグレーションを作り、外部キーと ON DELETE CASCADE を設定する
- モデルに 1 対多・多対多のリレーションと `$fillable` / `$casts` を定義する
- ファクトリとシーダーを用意し、`migrate:fresh --seed` で初期データを投入する

このセクションでは、9-1 の設計図をマイグレーションとモデルに落とし込み、初期データを入れて、データベースの土台を完成させます。

---

## 導入: 設計図を、動くテーブルとモデルにする

9-1 で、`users`・`categories`・`tags`・`tasks` の 4 実体と、`task_tag`・`favorites` の 2 ピボットを設計しました。9-2 で画面とルートを置きました。残るは、その画面が読み書きするデータの器、つまりテーブルとモデルです。

ここでは、外部キーとカスケード削除を正しく設定し、多対多のリレーションを `belongsToMany` で定義します。3-1 で学んだピボットの 2 パターン（純粋なピボット／一意制約を持つピボット）を、`task_tag` と `favorites` でそれぞれ実装します。最後にシーダーで初期データを入れ、画面で確認できる状態にします。

### 🧠 先輩エンジニアの思考プロセス

> マイグレーションで一番事故りやすいのが、外部キーの順序とカスケードです。`tasks` が `categories` を参照するなら、`categories` のマイグレーションが先に走らないと外部キーが張れません。`make:model` を作る順番がそのまま実行順になるので、参照される側（カテゴリ・タグ）から先に作るようにしています。カスケードは「親を消したら子も消す」を意図どおりに設定できているか、シーダーを入れた後に一度試して確かめます。

## 🏃 実践: テーブルとモデルを作る

### 🏃 Step 1: モデルとマイグレーションを生成する

モデルとマイグレーション（一部はファクトリも）を生成します。`-m` はマイグレーション、`-f` はファクトリを同時に作るオプションです。**参照される側（カテゴリ・タグ）から先に** 作り、続いて `tasks`、最後にピボット 2 つを作ります。この順序が、そのままマイグレーションの実行順になります。

```bash
# task-app ディレクトリで実行
sail artisan make:model Category -mf
sail artisan make:model Tag -mf
sail artisan make:model Task -mf
sail artisan make:migration create_task_tag_table
sail artisan make:migration create_favorites_table
```

これで、`Category` / `Tag` / `Task` のモデル・マイグレーション・ファクトリと、`task_tag` / `favorites` のマイグレーションが生成されます（`task_tag`・`favorites` はピボットなので、モデルは作りません）。

### 🏃 Step 2: マイグレーションを編集する

生成された 5 つのマイグレーションファイル（`database/migrations/` 配下）を、順に編集します。ファイル名の日時部分（`xxxx_xx_xx_xxxxxx`）は生成時刻によって変わるので、`create_xxx_table` の部分で見分けてください。

まず `categories` テーブル。`name` に一意制約を付けます。

```php
// database/migrations/xxxx_xx_xx_xxxxxx_create_categories_table.php
<?php

use Illuminate\Database\Migrations\Migration;
use Illuminate\Database\Schema\Blueprint;
use Illuminate\Support\Facades\Schema;

return new class extends Migration
{
    public function up(): void
    {
        Schema::create('categories', function (Blueprint $table) {
            $table->id();
            $table->string('name')->unique();
            $table->timestamps();
        });
    }

    public function down(): void
    {
        Schema::dropIfExists('categories');
    }
};
```

`tags` テーブルも同じ形です。

```php
// database/migrations/xxxx_xx_xx_xxxxxx_create_tags_table.php
<?php

use Illuminate\Database\Migrations\Migration;

            $table->id();
            $table->string('name')->unique();
            $table->timestamps();

        Schema::dropIfExists('tags');

```

`tasks` テーブルは、所有者（`user_id`）と分類（`category_id`）への外部キーを持ちます。`user_id` はカスケード削除（ユーザーを消したらそのタスクも消す）、`category_id` は外部キーの既定の動作（参照されているカテゴリは削除できない）にします。

```php
// database/migrations/xxxx_xx_xx_xxxxxx_create_tasks_table.php
<?php

use Illuminate\Database\Migrations\Migration;
use Illuminate\Database\Schema\Blueprint;
use Illuminate\Support\Facades\Schema;

return new class extends Migration
{
    public function up(): void
    {
        Schema::create('tasks', function (Blueprint $table) {
            $table->id();
            $table->foreignId('user_id')->constrained()->onDelete('cascade');
            $table->foreignId('category_id')->constrained();
            $table->string('title');
            $table->text('description')->nullable();
            $table->string('status')->default('pending');
            $table->date('due_date')->nullable();
            $table->timestamps();
        });
    }

    public function down(): void
    {
        Schema::dropIfExists('tasks');
    }
};
```

🔑 `category_id` の `->constrained()` には `onDelete` を付けていません。外部キーの既定の動作では、参照されているカテゴリ（タスクが紐づくカテゴリ）はデータベースのレベルでも削除できません。その手前で、アプリ側のガード（9-5）が「タスクが紐づくカテゴリは削除できません」と分かりやすく止めます。データベース制約とアプリのガードの二段構えで、整合性を守ります。

`task_tag` テーブルは、3-1 で学んだ **純粋なピボット** です。`task_id` と `tag_id` の複合主キーを持ち、`id` も `timestamps` も置きません。両方の外部キーをカスケード削除にします。

```php
// database/migrations/xxxx_xx_xx_xxxxxx_create_task_tag_table.php
<?php

use Illuminate\Database\Migrations\Migration;
use Illuminate\Database\Schema\Blueprint;
use Illuminate\Support\Facades\Schema;

return new class extends Migration
{
    public function up(): void
    {
        Schema::create('task_tag', function (Blueprint $table) {
            $table->foreignId('task_id')->constrained()->onDelete('cascade');
            $table->foreignId('tag_id')->constrained()->onDelete('cascade');
            $table->primary(['task_id', 'tag_id']);
        });
    }

    public function down(): void
    {
        Schema::dropIfExists('task_tag');
    }
};
```

`favorites` テーブルは、3-1 で学んだ **一意制約を持つピボット** です。`id` を持ち、`user_id` + `task_id` に一意制約を張って、同じユーザーが同じタスクを二重にお気に入りすることを防ぎます。

```php
// database/migrations/xxxx_xx_xx_xxxxxx_create_favorites_table.php
<?php

use Illuminate\Database\Migrations\Migration;
use Illuminate\Database\Schema\Blueprint;
use Illuminate\Support\Facades\Schema;

return new class extends Migration
{
    public function up(): void
    {
        Schema::create('favorites', function (Blueprint $table) {
            $table->id();
            $table->foreignId('user_id')->constrained()->onDelete('cascade');
            $table->foreignId('task_id')->constrained()->onDelete('cascade');
            $table->unique(['user_id', 'task_id']);
        });
    }

    public function down(): void
    {
        Schema::dropIfExists('favorites');
    }
};
```

### 🏃 Step 3: モデルにリレーションを定義する

4 つのモデルに、リレーションと `$fillable` を定義します。まず `app/Models/Category.php` を **まるごと書き換えます**。

```php
// app/Models/Category.php
<?php

namespace App\Models;

use Illuminate\Database\Eloquent\Factories\HasFactory;
use Illuminate\Database\Eloquent\Model;
use Illuminate\Database\Eloquent\Relations\HasMany;

class Category extends Model
{
    use HasFactory;

    protected $fillable = ['name'];

    public function tasks(): HasMany
    {
        return $this->hasMany(Task::class);
    }
}
```

`app/Models/Tag.php` を **まるごと書き換えます**。タグからタスクへは、`task_tag` を介した多対多です。

```php
// app/Models/Tag.php
<?php

namespace App\Models;

use Illuminate\Database\Eloquent\Factories\HasFactory;
use Illuminate\Database\Eloquent\Model;
use Illuminate\Database\Eloquent\Relations\BelongsToMany;

class Tag extends Model
{
    use HasFactory;

    protected $fillable = ['name'];

    public function tasks(): BelongsToMany
    {
        return $this->belongsToMany(Task::class, 'task_tag');
    }
}
```

`app/Models/Task.php` を **まるごと書き換えます**。タスクは、所有者（`user`）と分類（`category`）への `belongsTo`、タグへの `belongsToMany`（`task_tag`）、お気に入りしたユーザーへの `belongsToMany`（`favorites`）を持ちます。`due_date` は日付としてキャストします。

```php
// app/Models/Task.php
<?php

namespace App\Models;

use Illuminate\Database\Eloquent\Factories\HasFactory;
use Illuminate\Database\Eloquent\Model;
use Illuminate\Database\Eloquent\Relations\BelongsTo;
use Illuminate\Database\Eloquent\Relations\BelongsToMany;

class Task extends Model
{
    use HasFactory;

    protected $fillable = [
        'user_id',
        'category_id',
        'title',
        'description',
        'status',
        'due_date',
    ];

    protected $casts = [
        'due_date' => 'date',
    ];

    public function user(): BelongsTo
    {
        return $this->belongsTo(User::class);
    }

    public function category(): BelongsTo
    {
        return $this->belongsTo(Category::class);
    }

    public function tags(): BelongsToMany
    {
        return $this->belongsToMany(Tag::class, 'task_tag');
    }

    public function favoritedByUsers(): BelongsToMany
    {
        return $this->belongsToMany(User::class, 'favorites');
    }
}
```

🔑 `due_date` を `$casts` で `date` にすると、データベースの文字列が Carbon の日付オブジェクトとして扱え、Blade で `$task->due_date?->format('Y/m/d')` のように整形できます。2-2 で確認したとおり、Laravel 10 のキャストは `casts()` メソッドではなく `$casts` プロパティで書きます。

最後に `app/Models/User.php` を **まるごと書き換えます**。標準の `User` に、所有するタスク（`tasks`）と、お気に入りに登録したタスク（`favoriteTasks`、`favorites` を介した多対多）を加えます。

```php
// app/Models/User.php
<?php

namespace App\Models;

use Illuminate\Database\Eloquent\Factories\HasFactory;
use Illuminate\Database\Eloquent\Relations\BelongsToMany;
use Illuminate\Database\Eloquent\Relations\HasMany;
use Illuminate\Foundation\Auth\User as Authenticatable;
use Illuminate\Notifications\Notifiable;

class User extends Authenticatable
{
    use HasFactory, Notifiable;

    protected $fillable = [
        'name',
        'email',
        'password',
    ];

    protected $hidden = [
        'password',
        'remember_token',
    ];

    protected $casts = [
        'email_verified_at' => 'datetime',
        'password' => 'hashed',
    ];

    public function tasks(): HasMany
    {
        return $this->hasMany(Task::class);
    }

    public function favoriteTasks(): BelongsToMany
    {
        return $this->belongsToMany(Task::class, 'favorites');
    }
}
```

📝 `favoritedByUsers`（Task 側）と `favoriteTasks`（User 側）は、どちらも `favorites` ピボットを指す多対多です。タスクから見れば「お気に入りしたユーザーたち」、ユーザーから見れば「お気に入りのタスクたち」で、同じ関係を両側から見ています。

`database/seeders/FavoriteSeeder.php` を **まるごと書き換えます**。ランキングが分かりやすくなるよう、先頭のタスクほど多くのユーザーにお気に入りされるようにします。`syncWithoutDetaching` を使うので、二重登録のエラーは起きません。

```php
// database/seeders/FavoriteSeeder.php
<?php

namespace Database\Seeders;

use App\Models\Task;
use App\Models\User;
use Illuminate\Database\Seeder;

class FavoriteSeeder extends Seeder
{
    public function run(): void
    {
        $users = User::all();
        $tasks = Task::all();

        if ($users->isEmpty() || $tasks->isEmpty()) {
            return;
        }

        // 先頭のタスクほど多くのユーザーにお気に入りされるよう、登録するタスク数を変える
        foreach ($users as $index => $user) {
            $count = max($tasks->count() - $index, 1);
            $taskIds = $tasks->take($count)->pluck('id')->toArray();
            $user->favoriteTasks()->syncWithoutDetaching($taskIds);
        }
    }
}
```

`migrate:fresh` は全テーブルを削除してからマイグレーションをやり直し、`--seed` で `DatabaseSeeder` を実行します。出力に、`categories`・`tags`・`tasks`・`task_tag`・`favorites` の作成と、`Database\Seeders\DatabaseSeeder` の実行が並べば成功です。

> ⚠️ **よくあるエラー**: `SQLSTATE[HY000]: General error: 1215 Cannot add foreign key constraint`
>
> **原因**: 参照される側のテーブルより先に、参照する側のマイグレーションが走っている（例: `categories` より先に `tasks`）。
>
> **対処法**: マイグレーションファイルの日時部分（ファイル名の先頭）の順序を確認してください。Step 1 の生成順（`Category` → `Tag` → `Task` → `task_tag` → `favorites`）どおりなら、参照される側が先になります。順序が崩れている場合は、ファイル名の日時部分を手で調整して並べ替えます。
