## 導入: 画面のないもう 1 つの入口

ここまで作ってきたのは、人がブラウザで使う画面（HTML）でした。9-1 の設計どおり、このアプリにはもう 1 つ、プログラムが使う入口、公開 API（JSON）を用意します。

API は画面を持たないので、認証は扱わず（8-3 と同じ公開 API として作ります）、登録・更新では誰のタスクかを `user_id` で受け取ります。レスポンスは API Resource で整形し、検索・絞り込み・ページネーションを備え、見つからないときは JSON の 404 を返します。動作確認は、7-3 で導入した Postman で行います。

### 🧠 先輩エンジニアの思考プロセス

> 画面と API を両方持つアプリでは、コントローラの置き場所を最初に分けておくのが効きます。`App\Http\Controllers\TaskController`（画面）と `App\Http\Controllers\Api\V1\TaskController`（API）は名前は同じでも役割が違う。同じファイルに混ぜると、HTML を返すのか JSON を返すのかが曖昧になります。名前空間とディレクトリで最初から分けておくと、後から API のバージョンを増やすときも迷いません。

## 🏃 実践: 公開タスク API を実装する

### 🏃 Step 1: API Resource を作る

レスポンスを整形する 3 つの Resource を生成します。

```bash
# task-app ディレクトリで実行
sail artisan make:resource Api/V1/TaskResource
sail artisan make:resource Api/V1/CategoryResource
sail artisan make:resource Api/V1/TagResource
```

`app/Http/Resources/Api/V1/TaskResource.php` を **まるごと書き換えます**。外部に見せたいフィールドだけを並べ、カテゴリとタグはネストして返します。お気に入り数（`favorites_count`）も整形して返します。

```php
// app/Http/Resources/Api/V1/TaskResource.php
<?php

namespace App\Http\Resources\Api\V1;

use Illuminate\Http\Resources\Json\JsonResource;

class TaskResource extends JsonResource
{
    public function toArray($request)
    {
        return [
            'id' => $this->id,
            'title' => $this->title,
            'description' => $this->description,
            'status' => $this->status,
            'due_date' => $this->due_date?->format('Y-m-d'),
            'category' => new CategoryResource($this->whenLoaded('category')),
            'tags' => TagResource::collection($this->whenLoaded('tags')),
            'favorites_count' => (int) ($this->favorites_count ?? 0),
            'user_id' => $this->user_id,
        ];
    }
}
```

`app/Http/Resources/Api/V1/CategoryResource.php` を **まるごと書き換えます**。

```php
// app/Http/Resources/Api/V1/CategoryResource.php
<?php

namespace App\Http\Resources\Api\V1;

use Illuminate\Http\Resources\Json\JsonResource;

class CategoryResource extends JsonResource
{
    public function toArray($request)
    {
        return [
            'id' => $this->id,
            'name' => $this->name,
        ];
    }
}
```

`app/Http/Resources/Api/V1/TagResource.php` を **まるごと書き換えます**。

```php
// app/Http/Resources/Api/V1/TagResource.php
<?php

namespace App\Http\Resources\Api\V1;

use Illuminate\Http\Resources\Json\JsonResource;

class TagResource extends JsonResource
{
    public function toArray($request)
    {
        return [
            'id' => $this->id,
            'name' => $this->name,
        ];
    }
}
```

🔑 7-2 で学んだとおり、`category` は単一なので `new CategoryResource(...)`、`tags` は複数なので `TagResource::collection(...)` で包み、どちらも `whenLoaded` でくるみます。これで、コントローラで `with(['category', 'tags'])` を付けて読み込んだときだけネストされ、読み込み忘れによる N+1 を避けられます。`favorites_count` は `(int)` で整数にし、未集計のときは `?? 0` で 0 にします。

### 🏃 Step 2: API 用の FormRequest を作る

一覧（検索パラメータ）・登録・更新の検証を、専用の FormRequest に分けます。

```bash
# task-app ディレクトリで実行
sail artisan make:request Api/V1/IndexTaskRequest
sail artisan make:request Api/V1/StoreTaskRequest
sail artisan make:request Api/V1/UpdateTaskRequest
```

`app/Http/Requests/Api/V1/IndexTaskRequest.php` を **まるごと書き換えます**。クエリパラメータ（キーワード・状態・カテゴリ・ページ・件数）を検証します。`per_page` は 100 以下に制限します。

```php
// app/Http/Requests/Api/V1/IndexTaskRequest.php
<?php

namespace App\Http\Requests\Api\V1;

use Illuminate\Foundation\Http\FormRequest;

class IndexTaskRequest extends FormRequest
{
    public function authorize(): bool
    {
        return true;
    }

    public function rules(): array
    {
        return [
            'keyword' => ['nullable', 'string', 'max:255'],
            'status' => ['nullable', 'in:pending,in_progress,completed'],
            'category_id' => ['nullable', 'integer', 'exists:categories,id'],
            'page' => ['nullable', 'integer', 'min:1'],
            'per_page' => ['nullable', 'integer', 'min:1', 'max:100'],
        ];
    }
}
```

💡 8-1・8-3 では、`per_page` の上限を「大きすぎたら 100 に丸める（クランプ）」で扱いました。ここでは、`max:100` のバリデーションで「100 を超えたら 422 で弾く」方法を取ります。どちらも上限を守る有効なやり方です。クランプは黙って丸めるのに対し、バリデーションは「100 以下で指定してください」と利用側に明示的に伝える契約になります。本アプリの公開 API では、後者を採用します。

登録用の `app/Http/Requests/Api/V1/StoreTaskRequest.php` を **まるごと書き換えます**。公開 API なので、誰のタスクかを `user_id` で受け取ります（必須・存在するユーザー）。

```php
// app/Http/Requests/Api/V1/StoreTaskRequest.php
<?php

namespace App\Http\Requests\Api\V1;

use Illuminate\Foundation\Http\FormRequest;

class StoreTaskRequest extends FormRequest
{
    public function authorize(): bool
    {
        return true;
    }

    public function rules(): array
    {
        return [
            'user_id' => ['required', 'integer', 'exists:users,id'],
            'category_id' => ['required', 'integer', 'exists:categories,id'],
            'title' => ['required', 'string', 'max:255'],
            'description' => ['nullable', 'string'],
            'status' => ['required', 'in:pending,in_progress,completed'],
            'due_date' => ['nullable', 'date'],
            'tags' => ['nullable', 'array'],
            'tags.*' => ['integer', 'exists:tags,id'],
        ];
    }
}
```

更新用の `app/Http/Requests/Api/V1/UpdateTaskRequest.php` を **まるごと書き換えます** （検証内容は登録用と同じにします）。


### 🏃 Step 3: API コントローラを実装する

`Api\V1` 名前空間のコントローラを、`--api` で生成します。

```bash
# task-app ディレクトリで実行
sail artisan make:controller Api/V1/TaskController --api --model=Task
```

`app/Http/Controllers/Api/V1/TaskController.php` を **まるごと書き換えます**。一覧は検索・絞り込み・ページネーション、詳細・登録（201）・更新・削除（204）を実装します。一覧では `with` / `withCount`、詳細では `load` / `loadCount` を使い分けます（5-2）。

<details>
<summary>app/Http/Controllers/Api/V1/TaskController.php の全体（クリックで展開）</summary>

```php
// app/Http/Controllers/Api/V1/TaskController.php
<?php

namespace App\Http\Controllers\Api\V1;

use App\Http\Controllers\Controller;
use App\Http\Requests\Api\V1\IndexTaskRequest;
use App\Http\Requests\Api\V1\StoreTaskRequest;
use App\Http\Requests\Api\V1\UpdateTaskRequest;
use App\Http\Resources\Api\V1\TaskResource;
use App\Models\Task;

class TaskController extends Controller
{
    public function index(IndexTaskRequest $request)
    {
        $query = Task::with(['category', 'tags'])
            ->withCount(['favoritedByUsers as favorites_count']);

        if ($request->filled('keyword')) {
            $keyword = $request->input('keyword');
            $query->where(function ($q) use ($keyword) {
                $q->where('title', 'like', "%{$keyword}%")
                    ->orWhere('description', 'like', "%{$keyword}%");
            });
        }

        if ($request->filled('status')) {
            $query->where('status', $request->input('status'));
        }

        if ($request->filled('category_id')) {
            $query->where('category_id', $request->input('category_id'));
        }

        $perPage = $request->input('per_page', 20);
        $tasks = $query->latest()->paginate($perPage);

        return TaskResource::collection($tasks);
    }

    public function show(Task $task)
    {
        $task->load(['category', 'tags']);
        $task->loadCount(['favoritedByUsers as favorites_count']);

        return new TaskResource($task);
    }

    public function store(StoreTaskRequest $request)
    {
        $validated = $request->validated();

        $task = Task::create([
            'user_id' => $validated['user_id'],
            'category_id' => $validated['category_id'],
            'title' => $validated['title'],
            'description' => $validated['description'] ?? null,
            'status' => $validated['status'],
            'due_date' => $validated['due_date'] ?? null,
        ]);

        $task->tags()->sync($validated['tags'] ?? []);

        $task->load(['category', 'tags']);
        $task->loadCount(['favoritedByUsers as favorites_count']);

        return (new TaskResource($task))
            ->response()
            ->setStatusCode(201);
    }

    public function update(UpdateTaskRequest $request, Task $task)
    {
        $validated = $request->validated();

        $task->update([
            'user_id' => $validated['user_id'],
            'category_id' => $validated['category_id'],
            'title' => $validated['title'],
            'description' => $validated['description'] ?? null,
            'status' => $validated['status'],
            'due_date' => $validated['due_date'] ?? null,
        ]);

        $task->tags()->sync($validated['tags'] ?? []);

        $task->load(['category', 'tags']);
        $task->loadCount(['favoritedByUsers as favorites_count']);

        return new TaskResource($task);
    }

    public function destroy(Task $task)
    {
        $task->delete();

        return response()->json(null, 204);
    }
}
```

</details>

`index` では、`filled` でパラメータがあるときだけ条件を足し、キーワードはタイトルと説明の部分一致、状態とカテゴリは完全一致で絞ります。`store` は作成後に 201、`destroy` は本文なしの 204 を返します。`show` / `update` / `destroy` は、引数の `Task $task` に暗黙のルートモデルバインディングで該当タスクが入ります。

### 🏃 Step 4: API ルートを定義する

### 🏃 Step 5: 見つからないときの JSON 例外を設定する

存在しない ID にアクセスしたとき、JSON の 404 を返すようにします（8-2）。`app/Exceptions/Handler.php` を開き、内容を次のように **まるごと書き換えます** （`ModelNotFoundException` の `use` と `render` メソッドを加えています）。

```php
// app/Exceptions/Handler.php
<?php

namespace App\Exceptions;

use Illuminate\Database\Eloquent\ModelNotFoundException;
use Illuminate\Foundation\Exceptions\Handler as ExceptionHandler;
use Throwable;

class Handler extends ExceptionHandler
{
    /**
     * The list of the inputs that are never flashed to the session on validation exceptions.
     *
     * @var array<int, string>
     */
    protected $dontFlash = [
        'current_password',
        'password',
        'password_confirmation',
    ];

    /**
     * Register the exception handling callbacks for the application.
     */
    public function register(): void
    {
        $this->reportable(function (Throwable $e) {
            //
        });
    }

    public function render($request, Throwable $e)
    {
        if ($request->is('api/*') && $e instanceof ModelNotFoundException) {
            return response()->json([
                'error' => 'タスクが見つかりませんでした。',
            ], 404);
        }

        return parent::render($request, $e);
    }
}
```

これで、API（`api/*`）でモデルが見つからないときだけ、JSON の 404 を返します。Web 画面のエラー処理には影響しません。

### 🏃 Step 6: Postman で動作を確認する

7-3 で用意した Postman で、API を叩いて確認します。`base_url` 環境変数（`http://localhost`）を選び、新しいリクエストには **`Accept` ヘッダーに `application/json`** を設定します（これがないと、バリデーション失敗時に JSON ではなくリダイレクトが返ることがあります）。

**一覧（GET）** を確認します。

- メソッド: `GET`
- URL: `{{base_url}}/api/v1/tasks`
- Headers: `Accept` に `application/json`

`Send` すると、`data` にタスクが並び、`links` と `meta` が付いた JSON が返ります（9-3 のシーダーで 9 件入っています。`meta` は一部を抜粋しています）。

```json
{
  "data": [
    {
      "id": 9,
      "title": "部屋を片付ける",
      "description": null,
      "status": "completed",
      "due_date": "2026-06-08",
      "category": { "id": 2, "name": "プライベート" },
      "tags": [],
      "favorites_count": 1,
      "user_id": 3
    }
  ],
  "meta": {
    "current_page": 1,
    "last_page": 1,
    "per_page": 20,
    "total": 9
  }
}
```

**検索・絞り込み** を確認します。URL のクエリパラメータを変えて `Send` します。

- キーワード: `{{base_url}}/api/v1/tasks?keyword=メール` → タイトルに「メール」を含むタスク（「取引先にメールを返信する」）だけが返る
- 状態: `{{base_url}}/api/v1/tasks?status=completed` → 完了したタスクだけが返る
- カテゴリ: `{{base_url}}/api/v1/tasks?category_id=1` → カテゴリ「仕事」（id 1）のタスクだけが返る

**詳細（GET）** を確認します。

- URL: `{{base_url}}/api/v1/tasks/1`

カテゴリ・タグがネストされた 1 件のタスクが、`data` の下に返ります。

**登録（POST）** を確認します。

- メソッド: `POST`
- URL: `{{base_url}}/api/v1/tasks`
- Headers: `Accept` に `application/json`
- Body: `raw` → `JSON` を選び、次を入力

```json
{
  "user_id": 1,
  "category_id": 1,
  "title": "API から登録したタスク",
  "description": "Postman で作成しました",
  "status": "pending",
  "due_date": "2026-07-15",
  "tags": [1, 2]
}
```

`Send` すると、作成されたタスクが返り、ステータスは **201 Created** になります（Postman の画面右上にステータスコードが表示されます）。`tags` には指定した 2 つのタグがネストされます。

わざと不正な入力（タイトルなし・不正な状態・カテゴリなし）を送ると、**422** とバリデーションエラーが JSON で返ります。

```json
{
  "user_id": 1,
  "status": "done"
}
```

```json
{
  "message": "タイトルは必須です。 (and 2 more errors)",
  "errors": {
    "category_id": ["カテゴリは必須です。"],
    "title": ["タイトルは必須です。"],
    "status": ["選択された状態は正しくありません。"]
  }
}
```

**更新（PUT）と削除（DELETE）** を確認します。

- 更新: メソッド `PUT`、URL `{{base_url}}/api/v1/tasks/1`、Body に登録と同じ形の JSON（`title` などを変更）→ **200** と更新後のタスク
- 削除: メソッド `DELETE`、URL `{{base_url}}/api/v1/tasks/9` → **204 No Content** （本文は空。Postman でステータスコードを確認）

最後に、**異常時の応答** を確認します。

- 存在しない ID: `GET {{base_url}}/api/v1/tasks/99999` → **404** と `{"error": "タスクが見つかりませんでした。"}`
- 上限超過の件数: `GET {{base_url}}/api/v1/tasks?per_page=200` → **422** と `per_page` のバリデーションエラー（「1ページあたりの件数は100以下で指定してください。」）

これらがすべて期待どおりに返れば、公開 API は完成です。

💡 一覧・詳細・更新は 200、登録は 201、削除は 204、見つからないときは 404、検証エラーは 422、と HTTP ステータスコードで結果を伝えています（8-2）。利用側はこの番号を見て、成否を機械的に判断できます。
